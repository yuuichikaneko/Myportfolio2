# 非機能要件定義（PC構成提案Webアプリ）

版数: v1.0
最終更新日: 2026-03-21

## 1. 目的
本書は、機能要件を安定して提供するための非機能要件を定義する。対象は Web フロントエンド、Django API、データ収集バッチ、データベース運用とする。

## 2. 適用範囲
- フロントエンド（React）
- API（Django + DRF）
- データ収集（スクレイピング管理コマンド）
- データベース（PostgreSQL 前提）
- 監視、ログ、バックアップ、運用手順

## 3. 非機能要件

### 3.1 性能要件
- API `POST /api/generate-config` の応答時間:
  - 通常時 P95 <= 3.0 秒
  - P99 <= 5.0 秒
- API `GET /api/parts` の応答時間:
  - P95 <= 1.0 秒
- フロントエンド初回表示:
  - LCP <= 2.5 秒（主要画面、一般的なブロードバンド環境）
- 同時アクセス:
  - 100 同時ユーザーで上記 SLO を満たす。

### 3.2 可用性・信頼性要件
- 月間可用性目標: 99.0% 以上。
- 重大障害時の目標復旧時間（RTO）: 60 分以内。
- 目標復旧時点（RPO）: 24 時間以内。
- 外部データ取得失敗時は前回成功キャッシュでサービス継続する。
- スクレイピング失敗時は 3 回まで指数バックオフで再試行する。

### 3.3 拡張性要件
- 主要パーツカテゴリ追加時、既存 API 互換を維持しながら拡張可能であること。
- DB は正規化済みスキーマを前提とし、参照テーブル追加で属性拡張できること。
- 将来的な水平分割に備え、API 層はステートレスに維持する。

### 3.4 セキュリティ要件
- 全通信を HTTPS 必須とする（本番環境）。
- API 認証・認可:
  - 公開 API を除きトークン認証を適用する。
  - 管理系エンドポイントは権限分離する。
- 入力検証:
  - すべての入力値に型、範囲、列挙チェックを行う。
- 脆弱性対策:
  - SQL インジェクション、XSS、CSRF をフレームワーク標準機能で防御する。
  - 依存ライブラリの脆弱性を定期チェックする（月 1 回以上）。
- レート制限:
  - API に IP またはクライアント単位の制限を設定する。

### 3.5 運用・保守要件
- デプロイ:
  - 本番反映時はロールバック手順を事前定義する。
- 監視:
  - API 応答時間、エラー率、スクレイピング成功率、DB 接続エラーを監視する。
- アラート:
  - 連続 5 分でエラー率 > 5% の場合に通知する。
- ログ:
  - API アクセスログ、アプリケーションログ、バッチログを分離して保持する。
  - ログ保持期間は 90 日以上。

### 3.6 データ品質要件
- パーツデータ更新頻度: 1 日 1 回（06:00 JST 目安）。
- データ鮮度:
  - API で最終更新日時を返す。
- 異常値検知:
  - 価格 0 円、必須属性欠損、異常な重複データを検出し、除外または警告する。

### 3.7 バックアップ・災害対策要件
- DB バックアップ:
  - 毎日 1 回のフルバックアップを取得する。
- リストア検証:
  - 月 1 回以上、復元テストを実施する。
- 保管:
  - バックアップは本番環境と分離したストレージに保管する。

### 3.8 互換性・移行要件
- API 互換性:
  - 既存クライアントが利用する主要レスポンス項目を維持する。
- DB 移行:
  - SQLite から PostgreSQL への移行時は移行手順書と検証結果を残す。
- 文字コード:
  - UTF-8 を使用し、日本語データを正しく保持する。

### 3.9 テスト要件
- 自動テスト:
  - 主要ドメインロジックと API の回帰テストを CI で実行する。
- 品質ゲート:
  - 主要テスト失敗時はデプロイ不可。
- 性能試験:
  - リリース前に負荷試験を実施し、SLO 達成を確認する。

## 4. 受け入れ基準（非機能）
- NFR-01: `POST /api/generate-config` が通常負荷で P95 3 秒以内。
- NFR-02: 月間可用性 99.0% 以上。
- NFR-03: 障害時に RTO 60 分以内で復旧可能。
- NFR-04: HTTPS 強制、主要入力検証、レート制限が有効。
- NFR-05: 監視ダッシュボードで応答時間・エラー率・収集成功率を確認可能。
- NFR-06: 日次バックアップと月次リストア検証の記録が存在。

## 5. 改定ルール
- 本書の改定は機能追加、運用方針変更、インフラ変更時に実施する。
- 版数、改定日、変更概要を必ず更新する。
