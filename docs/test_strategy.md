# テスト方針（段階復帰運用）

## 1. 目的
設計変更（用途コード統一、互換変換、選定ロジック更新）に追従しつつ、
回帰検知の信頼性を保ったまま CI を段階的に強化する。

## 2. テスト分類
- 契約テスト（API 入出力）
  - 例: 用途コード変換、必須レスポンス項目、入力バリデーション
  - 失敗時の優先度: 最優先
- ロジックテスト（選定・重み・互換）
  - 例: 用途別ルール、予算帯挙動、互換コード正規化
  - 失敗時の優先度: 高
- 在庫依存テスト（データセット/在庫条件で結果変動）
  - 例: 固定商品名、固定価格、特定 SKU 前提
  - 失敗時の優先度: 中（期待値の保守が必要）

## 3. 現在のCI運用
- Push / PR の必須バックエンドテスト:
  - `python manage.py test scraper.tests.UsageConversionRegressionTests`
- 手動実行（workflow_dispatch）での全体確認:
  - `python manage.py test scraper.tests --verbosity 1`

## 4. 段階復帰プラン
- Phase 1（現行）
  - 契約テストを必須ゲートに固定
- Phase 2
  - `scraper.tests` 全体を手動/夜間で継続実行
  - FAIL の分類（仕様差分 or 実装バグ）を都度反映
- Phase 3
  - `scraper.tests` 全体を PR 必須ゲートへ昇格
  - 固定商品名依存のテストを閾値・ルールベースへ置換

## 5. 更新ルール
- 仕様変更時は、まず契約テストを更新する
- ロジック変更時は、用途別ルールテストを同時更新する
- 在庫依存テストは固定値を最小化し、
  - 価格: 範囲条件
  - 商品: カテゴリ/閾値条件
 へ寄せる

## 6. 既存ルート直下テストの扱い
- ルート直下 `test_*.py` は CI 必須対象から外し、
  運用スクリプトとして管理する
- CI の公式対象は `django/scraper/tests.py` 系を基準とする
