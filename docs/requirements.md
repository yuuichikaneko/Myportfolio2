# PC構成提案Webアプリ 要件定義書

## 1. 目的
ユーザーが予算と用途（ゲーミングPC、クリエイターPC、AI PC（ローカルAI）、汎用PC（事務・学習向け））を入力すると、互換性を考慮した最適なPC構成を自動提案し、購入リンクを表示する。

## 2. スコープ
本システムは以下を対象とする。

- 予算・用途に応じたPC構成自動提案
- 主要PCパーツの価格情報収集とキャッシュ
- 提案結果の可視化と購入導線提供

以下は将来拡張として扱う。

- ユーザー登録およびお気に入り保存
- 性能指標（PassMark、ベンチマーク）表示
- 在庫状況チェック

## 3. 機能要件

### 3.1 必須機能

#### 3.1.1 予算入力機能
- ユーザーは日本円で予算を入力できる。
- 入力値は整数値で扱い、最小値・最大値をバリデーションする。

#### 3.1.2 用途選択機能
- ユーザーは以下から用途を選択できる。
- `gaming`（ゲーミングPC）
- `creator`（クリエイターPC）
- `ai`（AI PC（ローカルAI））
- `general`（汎用PC（事務・学習向け））

#### 3.1.3 用途定義の考え方
- `gaming` は既存のゲーミング要件定義を維持し、本書での変更は最小限とする。
- `creator` は動画編集、画像編集、配信、軽量な 3D 制作を想定する。
- `ai` はローカル LLM 実行、画像生成、AI 開発・検証を想定する。
- `general` は事務作業、学習、Web 会議、ブラウジングなどの日常用途を想定する。
- `video_editing` は互換性維持のため、`creator` の下位互換またはエイリアスとして扱ってよい。
- 用途選択は、予算配分・CPU選定・GPU選定・メモリ容量・ストレージ容量・冷却要件に反映する。

#### 3.1.4 構成生成機能
- 予算配分アルゴリズムに基づき、パーツ候補を選定する。
- 互換性チェックを実施する。
- CPUソケット整合性
- メモリ規格整合性
- 電源容量余裕（推奨W数を満たす）
- ケースサイズ整合性
- `creator` は CPU、メモリ、NVMe SSD、GPU のバランスを重視する。
- `ai` は GPU VRAM、メモリ容量、電源、冷却を重視する。

#### 3.1.5 パーツ情報取得機能
- 価格ドットコムを主要データソースとして価格・商品情報を収集する。
- 必要に応じてAmazon/Rakutenの購入リンクを表示する。
- 取得データはDBにキャッシュし、API応答で再利用する。

#### 3.1.6 結果表示機能
- 提案構成として以下を表示する。
- CPU
- GPU
- メモリ
- ストレージ
- 電源
- ケース
- 価格合計を表示する。
- 各パーツの購入リンクを表示する。
- 用途に応じた推奨理由と、なぜその構成になったかの説明を表示する。

#### 3.1.7 用途別必須ルール（実装向け）
- `gaming` は現行ロジックを優先し、GPU性能閾値や既存のゲーミング最適化条件を維持する。
- `creator` はメモリ 32GB 以上を下限とし、メインストレージは NVMe SSD 1TB 以上を下限とする。
- `ai` は離散 GPU を必須とし、GPU VRAM を優先指標として選定する。
- `general` は価格効率を優先し、通常用途で過剰なハイエンド構成を避ける。

#### 3.1.8 用途別詳細要件（creator / ai / general）

`creator`（クリエイターPC）:
- 想定ワークロード:
  - 動画編集（FHD/4K）
  - 画像編集（RAW 現像含む）
  - 配信、軽量 3D 制作
  - ゲーム配信（プレイしながらの配信を含む）
- 予算帯ガイド:
  - 15 万円未満: 入門向け（FHD 編集中心）
  - 15 万円以上 30 万円未満: 標準向け（4K 編集の実用域）
  - 30 万円以上: 高負荷向け（重いエフェクトや同時作業を想定）
- パーツ選定ルール:
  - CPU はマルチスレッド性能を重視し、極端なゲーミング偏重を避ける。
  - メモリは 32GB を下限、4K 編集や同時作業が多い場合は 64GB を優先する。
  - ストレージは NVMe SSD 1TB 以上を下限とし、素材保管用の追加ストレージを推奨する。
  - GPU は制作支援とエンコード性能を加味して選定し、VRAM 8GB 以上を推奨下限とする。
  - ゲーム配信を想定する場合は、配信中の負荷分散とエンコード安定性を優先する。
  - 長時間作業を想定し、静音性と冷却のバランスを確保する。
- 返却時説明要件:
  - 採用理由として「編集負荷」「同時作業」「容量余裕」の 3 観点を短文で表示する。

`ai`（AI PC（ローカルAI））:
- 想定ワークロード:
  - ローカル LLM 推論
  - 画像生成
  - AI 開発・検証
- 予算帯ガイド:
  - 20 万円未満: 軽量モデル検証向け
  - 20 万円以上 40 万円未満: 実用推論向け
  - 40 万円以上: 複数用途並行・高負荷向け
- パーツ選定ルール:
  - 離散 GPU を必須とし、GPU VRAM を最優先指標として選定する。
  - VRAM は 8GB を最小要件、実運用では 12GB 以上を推奨とする。
  - システムメモリは 32GB を下限、推奨は 64GB 以上とする。
  - ストレージは NVMe SSD 1TB 以上を下限とし、モデル格納を考慮して 2TB 以上を推奨する。
  - 電源容量は構成見積消費電力に対して 20% 以上の余裕を持たせる。
  - 冷却は高負荷連続実行を前提に、温度上限超過を避ける構成を優先する。
- 返却時説明要件:
  - 採用理由として「VRAM 容量」「メモリ余裕」「連続実行安定性」を表示する。

`general`（汎用PC（事務・学習向け））:
- 想定ワークロード:
  - 文書作成、表計算、Web 会議、ブラウジング
  - 学習用途、軽量な開発
- 予算帯ガイド:
  - 8 万円未満: 最小構成
  - 8 万円以上 15 万円未満: 標準構成
  - 15 万円以上: 快適性重視構成
- パーツ選定ルール:
  - 価格効率を優先し、通常用途で不要なハイエンド GPU は選ばない。
  - iGPU で要件を満たす場合は dGPU を省略可能とする。
  - メモリは 16GB を下限、快適性重視時は 32GB を推奨する。
  - ストレージは NVMe SSD 500GB 以上を下限とし、推奨は 1TB とする。
  - 消費電力と静音性を重視し、運用コストを抑える。
- 返却時説明要件:
  - 採用理由として「コスト効率」「日常作業性能」「省電力性」を表示する。

- ユーザー登録・ログイン
- お気に入り構成の保存
- 性能指標（PassMark、各種ベンチマーク）表示

## 4. 非機能要件

詳細定義は [non_functional_requirements.md](non_functional_requirements.md) を参照。
テスト運用方針は [test_strategy.md](test_strategy.md) を参照。

### 4.1 パフォーマンス
- 構成生成APIの応答時間は3秒以内を目標とする（通常負荷時）。

### 4.2 スケーラビリティ
- 同時アクセス100ユーザー程度に耐える構成とする。

### 4.3 セキュリティ
- 通信はHTTPSを必須とする。
- API認証はJWTを採用する。

### 4.4 可用性
- 稼働率99%以上を目標とする。

## 5. 技術要件
- フロントエンド: React + TailwindCSS
- バックエンド: FastAPI（Python）
- DB: PostgreSQL（パーツ情報キャッシュ）
- スクレイピング: Python（BeautifulSoup、Requests）、必要時Playwright
- 構成生成アルゴリズム: Python（PuLP または OR-Tools）
- ホスティング候補:
- AWS（EC2 + RDS）
- Vercel + Supabase

## 6. インターフェース要件

### 6.1 UIフロー
1. ホーム画面
2. 予算入力フォーム
3. 用途選択
4. `構成を生成` ボタン押下
5. 結果画面（パーツ一覧 + 購入リンク）

### 6.2 API設計（初期案）

用途コード一覧（API 入力値）:
- `gaming`: ゲーミングPC
- `creator`: クリエイターPC
- `ai`: AI PC（ローカルAI）
- `general`: 汎用PC（事務・学習向け）

用途互換ルール:
- `video_editing` は `creator` の互換用コードとして受け付ける。
- `business` と `standard` は `general` の互換用コードとして受け付ける。
- API の正規値は `gaming`、`creator`、`ai`、`general` とする。

用途別リクエスト例:

```json
{
  "budget": 180000,
  "usage": "creator"
}
```

```json
{
  "budget": 350000,
  "usage": "ai"
}
```

```json
{
  "budget": 120000,
  "usage": "general"
}
```

用途別レスポンス例は、共通の `parts` 配列と `total_price` を返し、`usage` は選択した用途コードを返す。

#### POST `/generate-config`
入力例:

```json
{
  "budget": 100000,
  "usage": "gaming"
}
```

出力例:

```json
{
  "usage": "gaming",
  "budget": 100000,
  "total_price": 98700,
  "parts": [
    {
      "category": "cpu",
      "name": "Example CPU",
      "price": 24800,
      "url": "https://example.com/cpu"
    },
    {
      "category": "gpu",
      "name": "Example GPU",
      "price": 42800,
      "url": "https://example.com/gpu"
    }
  ]
}
```

## 7. 制約条件
- 価格ドットコムの利用規約を遵守する。
- スクレイピング頻度を制御する（レート制限、アクセス間隔、リトライ制御）。
- 金額表記・入力は日本円対応とする。
- PCパーツ情報は1日1回を目安に定期更新する。

## 8. 開発ステップ
1. 構成生成アルゴリズム試作（仮データ）
2. スクレイピング実装（CPU/GPUから開始しDB保存）
3. FastAPIでAPI構築
4. ReactでUI開発
5. 統合テストとデプロイ

## 9. 受け入れ条件（初版）
- 指定予算と用途で、互換性エラーのない構成が返ること。
- 返却構成の価格合計が予算を大幅超過しないこと。
- 各パーツに購入リンクが含まれること。
- 通常時に3秒以内でレスポンスを返すこと。
